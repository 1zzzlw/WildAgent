**伪代码**：
```typescript
function extractBlueprint(text: string): Blueprint | null {
  const match = text.match(/```json\s*([\s\S]*?)```/)
  if (!match) return null
  try { return JSON.parse(match[1]) }
  catch { return null }
}
```

## 二、中期扩展（Phase 6 → Phase 7）

### 2.1 P1：RAG 检索

**动机**：规范文档持续增加（新增构件类型、材质库、设计模式文档），全部塞进 context 会超出 token 限制，且无关文档会稀释 LLM 注意力。

**实现方案**：

```
storage/knowledge_base/
├── SPEC.md
├── PRIMITIVES.md
├── MATERIALS.md
├── BEHAVIORS.md
├── 中式建筑设计指南.md       # 未来新增
├── 建筑结构安全规范.md       # 未来新增
└── ...更多文档
```

**RAG 架构**：

```python
# app/spec/rag_loader.py（新建）

class RAGSpecLoader(SpecLoader):
    def __init__(self, retriever: BaseRetriever):
        self.retriever = retriever
    
    def load(self, query: str = "") -> str:
        """根据用户意图检索最相关文档片段"""
        if not query:
            return self._load_all()  # 无意图时加载摘要
        docs = self.retriever.get_relevant_documents(query)
        return self._format_docs(docs)
```

**需要的组件**（架构已预留）：
- `app/rag/document_loader.py`：文档切片和加载
- `app/rag/embedding.py`：文本向量化
- `app/rag/vector_store.py`：向量存储和检索
- `app/rag/retriever.py`：检索接口

**接入方式**：`agent_service.py` 改一行
```python
# 现在
self.spec_loader = FileSpecLoader(SPEC_PATHS)

# 以后
self.spec_loader = RAGSpecLoader(retriever)
```

**技术选型建议**：
- Embedding 模型：与 LLM 同厂商（降低 API 调用复杂度）
- 向量库：初期用 Chroma（轻量、Python 原生、零配置），后期换 PostgreSQL + pgvector
- 切片策略：按二级标题切（## 开头），每切片 500~1500 tokens

### 2.2 P1：LangGraph Pipeline 升级

**动机**：`create_agent` 对流程控制力弱——LLM 可能"忘记"调用校验工具。LangGraph 可以将生成和校验分成确定性节点。

**LangGraph 状态图**：

```text
用户消息
  → load_context         # 加载规范文档 + 场景上下文
  → generate_draft       # LLM 生成初稿 Blueprint
  → validate_spatial     # 确定性节点：跑所有 Python 校验函数（不走 LLM）
  → check_result         # 路由：有错误 → fix_issues / 无错误 → respond
  → fix_issues           # LLM 根据校验报告修正 Blueprint（可选循环）
  → respond              # 返回最终结果
```

**改动**：
- `app/agent/graph.py`：实现 `create_graph()` 函数
- `app/services/agent_service.py`：`self.agent` 替换为 `self.graph`
- 所有现有 tool 函数**零改动直接复用**

```python
# agent_service.py 改动对比

# 现在
self.agent = create_agent(model=self.llm, tools=tools, system_prompt=...)
result = await self.agent.ainvoke({"messages": [...]})

# 以后
self.graph = create_graph(llm=self.llm, tools=tools, prompt=...)
result = await self.graph.ainvoke({"user_message": message, ...})
```

**依赖**：需安装 `langgraph` 包

## 三、长期扩展（Phase 7+）

### 3.1 二次检测（截屏校验）

**流程**：
```text
用户点击「AI 检测」按钮
  → 前端截取当前 3D 视口
  → 截屏 + Blueprint JSON → 后端
  → 后端视觉 LLM 对比渲染结果与参数意图是否一致
  → 如有不一致 → 生成修正 patch
  → 前端应用修正
```

**改动**：
- 前端：`AIChatPanel.vue` 新增检测按钮 + 截屏逻辑
- 后端：新增 `api/validate.py` 端点（REST）+ `services/vision_service.py`
- LLM 需要支持视觉输入

### 3.2 MCP 集成

**动机**：Agent 通过 MCP 协议接入外部数据源。

| MCP Server | 提供数据 | 使用场景 |
|------------|---------|---------|
| 地形数据 | DEM 高程、坡度 | 建筑选址、地基设计 |

### 3.3 多 Agent 协作

**动机**：拆分为专职 Agent，避免单一 Agent 职责过重。

```text
Generator Agent    → 负责生成 Blueprint 初稿
Validator Agent    → 负责空间关系和语义校验
Design Advisor     → 负责风格一致性、美学建议
Orchestrator       → 协调上述 Agent，管理冲突
```

## 四、升级原则

每次扩展必须遵守以下原则：

1. **接口向后兼容**：`query() -> str` 签名永不改变
2. **Tool 纯函数化**：所有 tool 不依赖外部状态，谁调用都一样
3. **渐进替换**：每层独立升级，不联动修改（Spec Loader → Agent 编排 → Tools）
4. **先跑通再优化**：不因为"以后要升级"而把现在的简单方案做复杂
5. **代码可回滚**：每次升级只改一个模块，出问题可以单独回滚
