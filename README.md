# WildAgent

## 前言

WildAgent 是一个 AI 辅助的参数化 3D 建筑编辑器。用户只需用自然语言描述想要的建筑，系统即可自动生成 .wild 蓝图文件并通过内置渲染引擎实时呈现 3D 效果。用户还可以通过拖拽、参数调节和对话交互对模型进行精细调整。项目核心由三部分组成：AI 驱动的蓝图生成、确定性的参数化渲染引擎、基于 Three.js 的实时 3D 预览。

## 问题

1. **既然普通大模型在给定 .wild 规范文档后也能输出格式正确的 JSON 蓝图，为什么还需要专门开发一个 Agent？直接用一个 prompt 模板调用 LLM 不就行了吗？**

    ```bash
    单次 LLM 调用确实能生成一份 .wild 文件——如果你需要的只是"输入一句话 → 得到一个完整的建筑"。但实际使用场景远比这复杂：

    多轮对话修改。 用户不是一次性说清楚所有需求。典型流程是"建一座别墅"→"加个车库"→"车库改到侧面"→"屋顶换成歇山顶"→"整体风格改现代简约"。每一步都需要 Agent 理解修改意图、定位受影响构件、执行增量操作，而不是每次都重新生成整个文件。一个 prompt 模板做不到多轮上下文理解和增量修改。

    修改需要精准定位。 "把第三根柱子加粗"——Agent 需要先查场景里有哪些柱子、它们的 ID 是什么、当前参数是什么，然后精确修改那一根。普通 LLM 调用没有"查询场景状态"这步，只能瞎猜。

    多参数联动。 改一面墙的位置，连接的梁要跟着动、屋顶要重新适配、开口（门窗）要重新计算世界坐标。Agent 调用工具函数时，引擎的 resolver 自动处理这些联动，不需要 LLM 自己推导。

    Agent 不只是生成 JSON。 它还提供设计建议（"这种跨度用歇山顶会让屋顶显得过重，要不要试试攒尖顶？"）、一键整理（"帮我把柱子间距调均匀"）、解释设计决策（"为什么柱子选了 3.2 米？因为你的层高是 4 米，柱高/层高比通常取 0.8"）。这些是设计协作，不是文件生成。

    简单说： 如果你只需要一个 prompt 模板，那 Agent 确实不需要。但如果你想要一个能陪你迭代设计、理解修改意图、自动处理联动、给出设计建议的协作工具，Agent 就是必要的——它的价值不在"生成"，在"持续协作"。
    ```

2. **市面上的 AI 3D 生成工具（Meshy、Tripo 等）已经能做到文生 3D，普通大模型也能输出 Three.js 代码或 JSON 场景描述。WildAgent 跟它们有什么本质区别？这个项目的独特价值在哪里？**

    ```bash
    和 AI 3D 生成工具（Meshy / Tripo）的区别：

    市面上主流工具的底层逻辑是"AI 扩散模型直接生成三角网格"——神经网络从文字/图片推测每个顶点的坐标。产出的是一份 .glb 死文件，几十 MB，改不了参数，同一句话生成两次结果不同，也不知道哪堆三角形是"柱子"哪堆是"屋顶"。

    WildAgent 走的是另一条路：AI 不生成几何，AI 只填参数表，三角形的计算交给确定性的数学引擎。 产物是一份几 KB 的 .wild JSON——它是活的参数文件，每个构件有语义 ID，可以独立编辑、联动修改、版本管理。"永不删除"版本策略保证 v1.0 的蓝图在未来引擎中永远正确渲染。
    ```
    
    | 主流 AI 3D 工具    | WildAgent                    |                               |
    | ------------------ | ---------------------------- | ----------------------------- |
    | **AI 做什么**      | 直接产出顶点坐标             | 只产出参数值                  |
    | **几何精度**       | 不确定，可能有模糊和拓扑错误 | 精确到小数，100% 可复现       |
    | **产物**           | .glb 死文件（几十 MB）       | .wild 活参数（几 KB）         |
    | **可否参数化编辑** | 否（只能在 mesh 表面雕刻）   | 是（每个参数独立可调）        |
    | **确定性**         | 无（扩散随机性）             | 有（同一 .wild 永远同一结果） |
    
    ```bash
    和大模型直接输出 Three.js 代码的区别：
    
    让 LLM 直接写 Three.js 代码本质上和 ThreeJSON 是一回事——AI 在"拼积木"。它输出的代码是 new Box(1.2, 0.4, 0.8) 放在 (0, 0.5, 0) 位置。这有几个问题：(1) 没有建筑语义——不知道这是柱子还是墙；(2) 没有空间关系推导——墙角不会自动连接；(3) 没有材质系统——没有风化、苔藓等程序化效果；(4) 代码不可持久化为资产。
    
    WildAgent 的 .wild + 引擎方案：AI 输出的是"设计参数"，引擎负责从参数算出几何。墙体自动斜接、梁自动对齐柱顶、屋顶自动适配墙体——这些是引擎的 resolver.ts 做的，不是 AI 做的。AI 不需要操心几何，它只需要描述"这里有四面墙、四根柱子、一个重檐屋顶"。
    
    所以有意义吗？有意义。 主流工具解决的是"快速得到一个大概的 3D 模型"，WildAgent 解决的是"精确设计一份可编辑、可复用、永久兼容的数字建筑资产"。不是谁替代谁，是不同的需求场景。
    ```

3. **WildAgent 生成的 3D 模型够精细吗？项目是否局限于建筑类型？能否扩展到其他领域？**
    ```bash
    一、精细度问题：
    模型的精细度由三样东西决定：
    1.几何精度：这是引擎的强项。wild-core 的几何构建器从数学公式出发计算每个顶点——柱子的卷杀曲线（entasis）、屋檐的弧度、石墙的砖缝凹凸——全部精确到浮点数。几何可以无限细分（LOD），精度没有天花板。跟 AI 扩散模型对比：扩散模型的几何可能出现模糊、拓扑错误、面片混乱；引擎算出来的几何是数学上干净的，永远是封闭流形。在几何层面，引擎已经足够好。
    
    2.材质表现：这是当前需要加强但天花板很高的环节。引擎已有一套程序化材质系统（PBR 参数 + 风化/苔藓/边缘磨损 + 噪声纹理），但当前实现还比较基础（在 bakeProceduralColors 里做简单的噪声叠加）。提升路径清晰：多层级噪声叠加（Voronoi + Perlin）、程序化法线扰动（bump mapping）、环境反射采样。这些都是 shader 层面的事，架构已预留扩展点，只是需要更多开发投入。
    
    3.渲染后处理：当前 viewer 只开了基础阴影和 ACES 色调映射。加上 SSAO（环境光遮蔽）、Bloom（泛光）、HDR 环境光照后，画面质量会质变。这跟模型本身无关，是渲染层的增强，Three.js 都有现成方案。
    结论：几何精度已经够好，材质和渲染是后续优化重点，引擎架构已支持，不涉及底层改动。
    二、扩展性问题：
    项目当前不局限于建筑，未来的扩展路径是明确的。
    1.当前阶段： 10 种构件类型（wall / column / roof / beam / floor / opening / stair / furniture / dense_brick / body）覆盖了建筑和化身两个领域。但这不是终局——body.ts（化身）的存在本身就证明了引擎框架不限于建筑。
    
    2.短期扩展（构件级）： 在 wild-lang 规范中定义新构件参数 → 在 types.ts 加接口 → 在 geometry/ 下写一个构建器函数（约 100-200 行 TypeScript）→ 引擎即可识别。比如加"拱门"：定义 arch 的参数（span、rise、thickness、profile），实现 buildArch()。每次扩展只加一个文件，不改引擎核心。
    
    3.中期扩展（引擎通用化）： 将引擎从"硬编码 10 种构件"重构为"可注册的构件系统"。蓝图通过 meta.plugins 声明需要的构件类型，引擎动态加载对应的构建器。这样一来：
    
    引擎不再绑定任何特定领域——建筑、家具、园林、城市街区，只要有人写构件插件，引擎就能渲染
    第三方可以开发独立的构件包，不需要修改引擎核心代码
    Agent 加载插件的规范文档后，自动学会新构件的生成能力
    长期扩展（生态）： 如果通用化完成，.wild 从一个"建筑文件格式"变成"通用参数化 3D 资产格式"。引擎的材质系统（PBR + 效果层）、空间关系推导、程序化纹理、"永不删除"版本策略——这些能力对所有参数化 3D 物体都适用，不限于建筑。
    
    边界： 有机体（动物、人脸、自然地形）不适合这个框架——它们很难用少量参数精确描述，那是 AI 扩散模型的领地。除此之外，任何可以用参数精确描述的物体都适合这套系统。建筑恰好是这类物体中复杂度最高、最能验证框架能力的代表——先把它做好，再做其他的就是降维。
    ```
