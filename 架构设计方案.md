# WILD 建筑编辑器 - 从零开发最终架构设计方案

## 0. 文档目标

这是一份面向“从 0 开始开发前端和后端”的完整架构设计文档。

目标是：开发者或 AI 仅凭这一份文档，就能理解要做什么、保留什么、重建什么、各模块怎么通信、前后端目录怎么组织、接口怎么设计、开发顺序怎么推进。

本项目唯一保留并复用的现有核心资产是：

```text
wild-core 渲染/重建引擎
```

它负责把 `.wild` 蓝图重建为可渲染的几何数据。新项目不重新设计渲染引擎，不让 AI 生成 Three.js 几何算法，不让后端参与三角形计算。

新项目要从 0 开发的是：

- 前端建筑编辑器。
- 后端 Agent 服务。
- 前后端通信协议。
- 场景状态管理。
- `.wild` 编辑、校验、保存、导出工作流。
- AI 对话生成和修改 `.wild` 的工具链。

## 1. 产品定义

WILD 建筑编辑器是一个 AI 辅助的参数化 3D 建筑编辑器。

它不是“一句话生成死模型”的工具，而是一个可以持续编辑的建筑设计工作台：

```text
用户输入自然语言
  -> AI 生成 .wild 初稿或修改建议
  -> 编辑器展示 3D 结果
  -> 用户选择构件、拖拽、调参、确认或撤销
  -> .wild 文件持续作为唯一源文件
  -> wild-core 根据 .wild 确定性重建几何
```

核心原则：

| 原则                     | 说明                                            |
| ------------------------ | ----------------------------------------------- |
| `.wild` 是唯一场景源文件 | 所有建筑结构、材质、行为最终都落到 `.wild` JSON |
| 渲染引擎只复用不重做     | `wild-core` 是现有核心引擎，新系统只接入它      |
| AI 不直接生成三角形      | AI 只生成参数、构件、材质、patch                |
| 用户保留最终决策权       | 批量修改、删除、覆盖场景必须用户确认            |
| 手动编辑不走 Agent       | 拖拽、属性面板、场景树操作直接修改本地状态      |

## 2. 已有资产与边界

当前项目中已有：

```text
wild-lang/              WILD 语言规范、schema、types
wild-core/              现有渲染/重建引擎
wild-viewer/            现有查看器，可作为参考，不作为最终编辑器
lantu/                  示例 .wild 文件
```

### 2.1 必须保留

`wild-core` 的职责保持不变：

```text
输入: Blueprint(.wild JSON)
输出: ReconstructedEntity
      - meshes
      - materialParams
      - boundingBox
      - physics
      - scripts
      - animation
```

现有核心入口：

```ts
parseBlueprint(json: string): Blueprint
reconstructEntity(bp: Blueprint): Promise<ReconstructedEntity>
```

新前端只需要调用这两个入口，然后把 `MeshData` 转成 Three.js 对象。

### 2.2 可以参考但不直接作为最终产品

`wild-viewer` 当前是查看器，不是完整编辑器。它可以作为以下内容的参考：

- 如何加载 `.wild` 文件。
- 如何调用 `parseBlueprint()` 和 `reconstructEntity()`。
- 如何把 `MeshData` 转换成 Three.js `BufferGeometry`。
- 如何处理基础点击交互。

但最终编辑器需要从 0 设计状态管理、组件、AgentBridge、属性面板、构件库、撤销重做等能力。

### 2.3 不允许重做

新项目不做这些事：

- 不重写 `wild-core` 的几何生成算法。
- 不让后端渲染 `.wild`。
- 不让 Agent 输出 Three.js 代码。
- 不把 GLB、OBJ、ThreeJSON 作为主资产格式。
- 不把 UI 状态混入几何语义。

## 5. 核心数据模型

### 5.1 Blueprint

`.wild` 文件本质是一个 JSON Blueprint。

最小结构：

```json
{
  "meta": {
    "version": "1.0",
    "type": "building",
    "name": "未命名建筑"
  },
  "geometry": {
    "elements": []
  },
  "materials": {},
  "behaviors": {}
}
```

标准顶层字段：

| 字段        | 是否必需 | 说明                         |
| ----------- | -------- | ---------------------------- |
| `meta`      | 是       | 版本、类型、名称             |
| `geometry`  | 是       | 构件、模板、实例、批量放置   |
| `materials` | 否       | 材质库                       |
| `behaviors` | 否       | 物理、脚本、动画             |
| `editor`    | 否       | 编辑器私有元数据，不影响渲染 |

### 5.2 Geometry

```json
{
  "geometry": {
    "elements": [],
    "templates": {},
    "instances": [],
    "placements": []
  }
}
```

标准构件类型：

| type          | 用途             |
| ------------- | ---------------- |
| `wall`        | 墙体             |
| `floor`       | 地板/平台        |
| `column`      | 柱子             |
| `beam`        | 梁               |
| `roof`        | 屋顶             |
| `opening`     | 门窗洞口         |
| `stair`       | 楼梯             |
| `furniture`   | 参数化家具和瓦片 |
| `dense_brick` | 高分辨率体素细节 |
| `body`        | 化身身体         |

### 5.3 SceneDocument

前端编辑器内部不只保存 Blueprint，还要保存编辑会话状态。

```ts
interface SceneDocument {
  id: string
  name: string
  revision: number
  blueprint: Blueprint
  dirty: boolean
  savedAt?: number
}
```

`revision` 是前后端协作的关键。每次成功应用 patch 后递增。

### 5.4 Editor Metadata

不影响引擎输出的编辑器信息放在 `blueprint.editor`：

```json
{
  "editor": {
    "version": "1.0",
    "groups": {
      "main_house": {
        "name": "主体建筑",
        "elements": ["wall_front", "wall_back", "main_roof"]
      }
    },
    "view": {
      "camera": {
        "position": [12, 10, 12],
        "target": [0, 1.5, 0]
      }
    },
    "agent": {
      "generatedBy": "wild-server",
      "lastPrompt": "生成一座中式凉亭"
    }
  }
}
```

硬规则：

- `editor` 不能影响 `wild-core` 的重建结果。
- `wild-core` 不能读取 `editor` 字段。
- 导出纯资产时可以剥离 `editor`。

## 6. ScenePatch 协议

所有编辑操作，无论来自用户还是 Agent，都转成 `ScenePatch`。

### 6.1 Patch 基本结构

```ts
interface ScenePatch {
  type: 'scene_patch'
  patch_id: string
  base_revision: number
  source: 'user' | 'agent' | 'system'
  mode: 'apply' | 'proposal'
  requires_confirmation: boolean
  operations: SceneOperation[]
  summary?: string
}
```

### 6.2 操作类型

```ts
type SceneOperation =
  | { op: 'add_element'; element: GeometryElement }
  | { op: 'update_element'; id: string; changes: Record<string, unknown> }
  | { op: 'remove_element'; id: string }
  | { op: 'upsert_material'; name: string; material: MaterialDef }
  | { op: 'add_template'; name: string; template: GeometryElement }
  | { op: 'update_template'; name: string; changes: Record<string, unknown> }
  | { op: 'remove_template'; name: string }
  | { op: 'add_instance'; instance: InstanceRef }
  | { op: 'add_placement'; placement: Placement }
  | { op: 'set_behaviors'; changes: Record<string, unknown> }
  | { op: 'set_editor_meta'; changes: Record<string, unknown> }
```

### 6.3 应用流程

```text
收到 patch
  -> 检查 base_revision 是否等于当前 revision
  -> 校验 operations 格式
  -> 应用到 Blueprint 副本
  -> 运行 Blueprint 校验
  -> 调用 wild-core reconstructEntity 做 smoke test
  -> 成功后替换 SceneStore.blueprint
  -> revision + 1
  -> 渲染刷新
  -> 记录 undo 快照
```

### 6.4 确认规则

| 操作               | 是否需要确认   |
| ------------------ | -------------- |
| 属性面板单字段修改 | 否             |
| 拖拽添加构件       | 否             |
| Agent 添加普通构件 | 可配置，默认否 |
| Agent 批量添加     | 是             |
| 删除构件           | 是             |
| 覆盖整个场景       | 是             |
| 一键整理           | 是             |
| 风格迁移           | 是             |

### 7.2 前端 Store

#### sceneStore

职责：

- 保存当前 `SceneDocument`。
- 应用 `ScenePatch`。
- 调用重建。
- 管理 dirty 状态。

核心状态：

```ts
interface SceneStoreState {
  document: SceneDocument | null
  reconstructed: ReconstructedEntity | null
  validationIssues: ValidationIssue[]
  isReconstructing: boolean
}
```

核心方法：

```ts
loadBlueprint(bp: Blueprint): void
applyPatch(patch: ScenePatch): Promise<void>
reconstruct(): Promise<void>
validate(): ValidationIssue[]
exportWild(): string
```

#### selectionStore

职责：

- 当前选中构件。
- hover 构件。
- 多选。

```ts
interface SelectionStoreState {
  selectedIds: string[]
  hoveredId: string | null
}
```

#### historyStore

职责：

- 撤销。
- 重做。
- patch 历史。

```ts
interface HistoryEntry {
  label: string
  before: Blueprint
  after: Blueprint
  patch: ScenePatch
}
```

#### agentStore

职责：

- 聊天消息。
- WebSocket 状态。
- Agent 正在执行的步骤。
- 等待用户确认的 patch。

### 7.3 渲染适配层

前端不能把渲染转换逻辑散落在 Vue 组件里，必须集中在 `renderer/`。

```text
renderer/wildCoreAdapter.ts
  - import wild-core
  - parseBlueprintText()
  - reconstructBlueprint()

renderer/meshDataToGeometry.ts
  - MeshData -> THREE.BufferGeometry

renderer/materialAdapter.ts
  - MaterialParams -> THREE.Material

renderer/renderEntity.ts
  - ReconstructedEntity -> THREE.Group

renderer/picking.ts
  - raycast result -> elementId
```

重建流程：

```text
SceneStore.blueprint
  -> wildCoreAdapter.reconstructBlueprint()
  -> ReconstructedEntity
  -> renderEntity()
  -> CanvasViewport 替换场景 group
```

MVP 可以整场重建。后期再做局部重建和缓存。

### 7.4 属性面板

属性面板根据选中构件的 `type` 动态渲染。

示例：

| 构件        | 控件                                                  |
| ----------- | ----------------------------------------------------- |
| `wall`      | from/to 坐标、厚度、材质、曲线                        |
| `column`    | base、height、bottomRadius、topRadius、style、材质    |
| `roof`      | roofType、span、depth、height、thickness、tiers、材质 |
| `opening`   | parentWall、from、width、height、style、材质          |
| `furniture` | subtype、position、dimensions、style、材质            |

属性变更时生成 patch：

```ts
{
  op: 'update_element',
  id: 'col_03',
  changes: { height: 3.8 }
}
```

### 7.5 构件库

构件库分两类：

```text
预设构件
  - wall
  - floor
  - column
  - beam
  - roof
  - opening
  - stair
  - furniture

AI 生成模块
  - 凉亭柱网
  - 门楣模板
  - 瓦片 placement
  - 中式屋顶组合
```

拖拽到视口时：

```text
drag start
  -> 创建构件草稿
drop on viewport
  -> 根据拾取点计算位置
  -> 生成 add_element 或 add_instance patch
  -> applyPatch
```

## 8. 后端架构

后端只负责 Agent、会话、保存、校验，不负责渲染。

### 8.1 服务模块

```text
FastAPI app
├── WebSocket Agent API       ← 传输层：收发消息，不碰 AI 逻辑
│   └── ws_heartbeat.py       ← 心跳监控工具类（超时检测 + 回调通知，与具体端点解耦）
├── Agent Service             ← 服务层：管理 Agent 生命周期，对外暴露 query()
├── LangGraph Agent           ← 编排层：定义"思考步骤"（graph + state + prompts）
├── Model Client              ← 工厂层：纯 LLM 初始化，无副作用
├── Tool Layer                ← 工具层：读写场景、生成 patch
├── Scene REST API
├── Validate REST API
├── Session Service
├── Scene Storage
└── Validation Service
```

**Agent 相关模块的分层关系**（从外到内，层层升级）：

```text
api/ws_agent.py          ← 传输：收 WebSocket 消息 → 调 AgentService → 发回前端
    ↓
services/agent_service.py ← 服务：组装 agent/ 各模块，对外暴露 query()，对内决定调用方式
    ↓
agent/graph.py           ← 编排：LangGraph 状态图，load_context → classify → inspect → plan → validate → respond
    ↓
agent/model_client.py    ← 工厂：create_llm()，纯函数，无全局变量
```

**MVP 阶段 `agent_service.py` 内部实现**：

```text
┌─ AgentService.__init__() ──────────────────────────────────────────┐
│                                                                     │
│  create_llm()          ← 调 model_client，拿到裸 LLM               │
│       ↓                                                            │
│  create_agent(          ← LangChain 包装：llm + tools + system_prompt │
│      model=llm,                          → 变成能对话的 agent       │
│      tools=[],                                                      │
│      system_prompt=...                                              │
│  )                                                                  │
│       ↓                                                            │
│  self.agent            ← 复用此实例，query() 每次调 agent.ainvoke() │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**升级路径**：MVP 用 `create_agent` 包一层就够；Phase 4 把 `create_agent` 替换为 LangGraph `graph`，`query()` 内部从 `agent.ainvoke()` 改为 `graph.ainvoke()`，`ws_agent.py` 一行不动。

#### 心跳协议

前端每 15s 发送 ping，后端立即回复 pong。后端通过 `WebSocketHeartbeat` 工具类监控：
空闲超过 90s 未收到任何消息 → 发送 `network_error` → 关闭连接。AI 处理期间
（`is_processing=True`）豁免超时检查。

```json
// 前端 → 后端
{ "type": "ping", "timestamp": 1711234567890 }

// 后端 → 前端
{ "type": "pong", "timestamp": 1711234567890 }

// 后端 → 前端（超时通知）
{ "type": "network_error", "error": "心跳超时，连接已断开", "reason": "heartbeat_timeout" }
```

前端连接管理：
- 指数退避自动重连（1s → 2s → 4s → ... → 30s 上限，±25% 抖动）
- 最大重试 10 次，超出后提示用户刷新
- 监听页面可见性变化（`visibilitychange`）：页面恢复可见时立即检测连接并补发心跳，
  解决浏览器后台标签页对 `setInterval` 的节流问题

## 9. Agent 架构

当前阶段使用单 Agent，不使用多 Agent。

### 9.1 原因

当前任务域集中在一份 `.wild` 文件上：

- 生成建筑。
- 添加构件。
- 修改参数。
- 生成模板。
- 给设计建议。
- 校验合理性。

这些不需要多个 Agent 协作。多 Agent 会增加通信、冲突合并、调试和延迟成本。

### 9.2 LangGraph 状态图

```text
用户消息
  -> load_context
  -> classify_intent
  -> inspect_scene
  -> plan_patch
  -> validate_patch
  -> respond
```

节点说明：

| 节点              | 职责                                     |
| ----------------- | ---------------------------------------- |
| `load_context`    | 读取用户消息、场景摘要、当前选择         |
| `classify_intent` | 判断是新建、添加、修改、删除、建议、解释 |
| `inspect_scene`   | 需要时调用读取工具获取元素详情           |
| `plan_patch`      | 生成完整场景或 ScenePatch                |
| `validate_patch`  | 校验 patch 是否符合规则                  |
| `respond`         | 返回自然语言说明和 patch                 |

### 9.3 Agent State

```py
class AgentState(TypedDict):
    session_id: str
    request_id: str
    scene_id: str | None
    scene_revision: int
    user_message: str
    scene_summary: dict
    selection: list[str]
    messages: list[dict]
    intent: str | None
    pending_patch: dict | None
    validation_issues: list[dict]
    final_reply: str | None
```

### 9.4 工具清单

读工具：

| 工具               | 输入          | 输出         |
| ------------------ | ------------- | ------------ |
| `inspect_scene`    | scene_summary | 场景概览     |
| `list_elements`    | type?         | 元素摘要列表 |
| `get_element_info` | element_id    | 元素完整参数 |
| `list_materials`   | 无            | 材质列表     |
| `get_selection`    | 无            | 当前选中元素 |

写工具：

| 工具                     | 输入         | 输出                              |
| ------------------------ | ------------ | --------------------------------- |
| `generate_scene`         | description  | 完整 Blueprint 草稿               |
| `propose_add_element`    | type, params | add_element patch                 |
| `propose_update_element` | id, changes  | update_element patch              |
| `propose_remove_element` | id           | remove_element patch              |
| `generate_module`        | description  | template/instance/placement patch |
| `organize_scene`         | scope        | 批量整理 patch                    |
| `suggest_design`         | focus        | 文本建议                          |
| `validate_patch`         | patch        | 校验结果                          |

### 9.5 Agent 行为规则

Agent 必须遵守：

- 所有场景修改都输出 `ScenePatch`。
- 不直接输出 Three.js、HTML、CSS、JS。
- 不生成未知构件类型。
- 不修改 `wild-core`。
- 不生成大段不可校验 JSON。
- 不清楚用户指代时，如果有 selection 就使用 selection，没有 selection 就提问。
- 删除、覆盖、批量整理必须设置 `requires_confirmation: true`。

### 9.6 Agent 蓝图生成与校验流水线（✨ 已实现）

Agent 生成蓝图的完整流水线由三层可替换模块组成：

```text
┌──────────────────────────────────────────────────────┐
│  ws_agent.py          ← 传输层，永不改动              │
│  agent_service.query() ← 服务层，接口不变              │
└──────────────┬───────────────────────────────────────┘
               │
    ┌──────────┴──────────┐
    │   Spec Loader       │  ← 可替换：os.read → RAG
    │   加载规范文档        │     FileSpecLoader → RAGSpecLoader
    └──────────┬──────────┘
               │
    ┌──────────┴──────────┐
    │   Agent 编排         │  ← 可替换：简单Agent → LangGraph
    │   调用 LLM + Tools    │     create_agent → graph.ainvoke
    └──────────┬──────────┘
               │
    ┌──────────┴──────────┐
    │   Validation Tools   │  ← 不改：纯函数，谁调用都一样
    │   空间校验/自动修正    │     5 检测 + 1 修正
    └──────────────────────┘
```

#### 9.6.1 Spec Loader（规范文档加载层）

**位置**：`app/spec/loader.py`

抽象基类 `SpecLoader` 定义了两个方法：
- `load() -> str`：加载全部规范文档，返回拼接文本
- `list_sources() -> list[str]`：返回来源列表

**当前实现**：`FileSpecLoader` —— 用 `pathlib.Path.read_text()` 直接读取文件

```python
# 现在：文件读取
loader = FileSpecLoader(["SPEC.md", "PRIMITIVES.md", "MATERIALS.md", "BEHAVIORS.md"])
spec_text = loader.load()  # ~24K 字符，远小于任何现代 LLM 上下文窗口
```

**升级到 RAG**：只需实现 `RAGSpecLoader(SpecLoader)`，替换一行代码：

```python
# 以后：向量检索
loader = RAGSpecLoader(retriever)
spec_text = loader.load(query="中式凉亭")  # 根据意图检索相关片段
```

#### 9.6.2 Validation Tools（空间校验工具集）

**位置**：`app/tools/spatial_tools.py`

所有工具都是 `@tool` 装饰的纯函数。因为纯函数不依赖任何外部状态，所以 Agent 升级到 LangGraph 时零改动直接复用。

| 工具 | 类型 | 功能 |
|------|------|------|
| `validate_blueprint_structure` | 检测 | 检查 meta/geometry/elements 结构完整性、ID 唯一性、type 字段 |
| `validate_opening_coords` | 检测 | 检查 opening 的 from 是否使用沿墙距离（非世界坐标） |
| `validate_wall_junctions` | 检测 | 检查相邻墙端点是否对齐（15cm 容差），防止墙体缝隙 |
| `validate_roof_coverage` | 检测 | 检查屋顶 span/depth 是否合理覆盖墙体范围 |
| `validate_stair_alignment` | 检测 | 检查楼梯端点高度是否与地板/墙体对齐 |
| `fix_opening_coords` | 修正 | 自动将世界坐标换算为沿墙距离 |

**扩展方式**：新增函数 `def validate_xxx(bp: dict) -> str`，在 `agent_service.py` 的 `tools` 列表注册即可。

#### 9.6.3 System Prompt 组装

**位置**：`app/agent/prompts.py`

`build_system_prompt(spec_text)` 将规范文档 + 空间规则 + 工作流程 + 输出格式组装成完整的 System Prompt。

核心规则（铁律）：
1. opening 的 from = [沿墙距离, 距墙底高度, 法向偏移]，禁止使用世界坐标
2. 墙体转角必须共享端点坐标，不得有缝隙
3. 屋顶 span/depth 必须覆盖被支撑墙体范围
4. 楼梯端点高度必须对齐地板或墙体

#### 9.6.4 完整生成流程

```text
用户: "建一个别墅"
  ↓
ws_agent.py                     ← 不动，接收 WebSocket 消息
  ↓
agent_service.query()           ← 不动，返回文本
  ↓
SpecLoader.load()               ← 加载规范文档注入 System Prompt
  ↓
Agent(LLM + 6 tools).ainvoke()
  │  LLM 分析需求 → 生成初稿
  │  → validate_blueprint_structure → "结构完整（12个构件）"
  │  → validate_opening_coords → "⚠️ window_front: from[0]=-5 超出墙体"
  │  → fix_opening_coords → "🔧 修正为沿墙距离 3.00"
  │  → validate_wall_junctions → "✅ 所有端点已对齐"
  │  → validate_roof_coverage → "✅ 屋顶尺寸合理"
  │  → validate_stair_alignment → "✅ 楼梯高度一致"
  ↓
返回最终 Blueprint JSON
  ↓
ws_agent 推给前端 → 前端 applyPatch → wild-core 重建 → 渲染
```

#### 9.6.5 升级路径一览

| 组件 | 现在 | 以后 | 改动量 |
|------|------|------|--------|
| Spec Loader | `FileSpecLoader` (os.read) | `RAGSpecLoader` (向量检索) | `agent_service.py` 改一行 |
| Agent 编排 | `create_agent` (LangChain) | `graph.ainvoke` (LangGraph) | `agent_service.py` 改内部实现 |
| Validation Tools | 6 个纯函数 | 不变（直接复用） | 零改动 |
| System Prompt | `build_system_prompt()` | 拆成多节点小 prompt | `prompts.py` 扩展 |
| 传输层 | `ws_agent.py` | 不变 | 零改动 |

## 10. 校验体系

### 10.1 前端校验

前端在应用 patch 前执行：

```text
patch 格式校验
  -> revision 校验
  -> Blueprint schema 校验
  -> 语义校验
  -> wild-core 重建 smoke test
```

语义校验包括：

- 构件 id 唯一。
- 材质引用存在。
- `opening.parentWall` 存在且是 `wall`。
- `behavior.constraints.target` 存在。
- `placement.template` 存在。
- 数值字段在合理范围。

### 10.2 后端校验

后端校验用于提前过滤 Agent 错误，但最终是否应用仍由前端决定。

后端需要和前端共用同一套 schema 来源：`wild-lang/schema.json`。

## 15. 最终架构总结

最终系统由三部分组成：

```text
1. wild-core
   已有渲染/重建引擎。
   不重做，不由 Agent 修改。

2. wild-editor
   从 0 开发的前端编辑器。
   负责 UI、状态、patch、调用 wild-core、渲染 Three.js。

3. wild-server
   从 0 开发的后端 Agent 服务。
   负责理解用户意图，生成 .wild Blueprint 或 ScenePatch。
```

最重要的数据流：

```text
用户操作 / Agent 建议
  -> ScenePatch
  -> SceneStore.blueprint
  -> wild-core.reconstructEntity()
  -> ReconstructedEntity
  -> Three.js 渲染
```

最重要的边界：

```text
Agent 只改参数，不生成几何。
Editor 只管状态，不发明语言。
wild-core 只管重建，不管 UI 和 AI。
.wild 永远是唯一源文件。
```

只要遵守这四条，前端和后端可以从 0 开发，现有渲染引擎可以稳定复用，后续再逐步扩展 MCP、RAG、局部重建和多格式导出。

### 16.4 扩展的核心原则

无论怎么扩展，以下四条不变：

1. **Spec Loader 接口不变**：`load() -> str`，换 RAG 只改一行
2. **Tool 函数签名不变**：`(blueprint: dict) -> str`，谁调用都一样

4. **分层边界不变**：传输 → 服务 → 编排 → 工具，每层只替换不重做

### 17.1 规范文档拆分（已完成）

**背景**：原 `BLUEPRINT-SPECIFICATION.md` 约 24K 字符，全量注入 System Prompt 对小模型压力过大，context 空间被挤占，生成质量下降。

**决策**：把规范文档拆成两个版本，按阶段使用不同版本：

| 文件 | 路径 | 用途 | 大小 |
|------|------|------|------|
| `BLUEPRINT-SPEC-MINIMAL.md` | `storage/knowledge_base/` | 当前使用，直接注入 System Prompt | ~5K 字符 |
| `BLUEPRINT-SPEC-FULL.md` | `storage/knowledge_base/` | 接入 RAG 后使用，分片建索引 | ~24K 字符 |
| `BLUEPRINT-SPECIFICATION.md` | `storage/knowledge_base/` | 原始文件，保留备份 | ~24K 字符 |

**MINIMAL 版本包含**：
- 10 种构件的必填字段 + 最小 JSON 示例
- 材质 PBR 格式速查（含常用预设值）
- 完整小屋示例（4墙 + 地板 + 2门窗 + 屋顶，含 opening 坐标验算）
- 板凳示例（用 column + floor 组合替代 furniture）

**当前引用**：`agent_service.py` 的 `SPEC_PATHS` 指向 `BLUEPRINT-SPEC-MINIMAL.md`

**后续升级（RAG 接入时）**：
1. 对 `BLUEPRINT-SPEC-FULL.md` 按章节分片（chunk），每片约 500~1000 tokens
2. 实现 `RAGSpecLoader(SpecLoader)`，接收用户 query，向量检索最相关片段
3. `agent_service.py` 中将 `FileSpecLoader` 替换为 `RAGSpecLoader`，一行改动，接口不变

### 17.2 Function Calling（工具调用）现状说明

**说明**：项目中已经在使用 Function Calling（即 LangChain 的 `@tool` 装饰器机制）。`spatial_tools.py` 里 7 个工具函数注册给 LangChain `create_agent` 后，框架会把它们的 schema 作为 tools 列表传给 LLM，LLM 自主决定调用哪些工具、传什么参数。

**Function Calling 的两个作用**：
1. 让 LLM 决定"何时调用工具、传什么参数"（主要用途）
2. 规范工具输入的格式（副作用）

**当前问题**：小模型的 tool use 可靠性差——不一定会主动调用全部 6 个校验工具，或调用顺序错误，导致带有空间错误的蓝图被直接输出。

**待做优化（服务端驱动的二阶段校验）**：

把"LLM 自愿调用工具校验"改成"服务端强制执行校验"，流程变为：

```text
阶段1：纯生成
  prompt 只聚焦生成，不带工具列表
  → LLM 输出 Blueprint JSON 初稿

阶段2：服务端校验（确定性 Python）
  agent_service 直接调用 spatial_tools 的底层函数（不经过 LLM）
  → 收集所有校验问题

阶段3：LLM 修正（如有问题）
  把校验结果拼入 prompt，让 LLM 针对性修正
  → 输出最终 Blueprint
```

改动范围：`agent_service.py` 内部实现，`query()` 接口和 `ws_agent.py` 不动。
实现位置：待 `agent/graph.py`（LangGraph）完成后一并改造，MVP 阶段暂保持现状。
