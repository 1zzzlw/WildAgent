# WILD 建筑编辑器 - 从零开发最终架构设计方案

## 0. 文档目标

这是一份面向“从 0 开始开发前端和后端”的完整架构设计文档。

目标是：开发者或 AI 仅凭这一份文档，就能理解要做什么、保留什么、重建什么、各模块怎么通信、前后端目录怎么组织、接口怎么设计、开发顺序怎么推进。

本项目唯一保留并复用的现有核心资产是：

```text
wild-core 渲染/重建引擎
```

它负责把 `.wild` 蓝图重建为可渲染的几何数据。新项目不重新设计渲染引擎，不让 AI 生成 Three.js 几何算法，不让后端参与三角形计算。

新项目要从 0 开发的是：

- 前端建筑编辑器。
- 后端 Agent 服务。
- 前后端通信协议。
- 场景状态管理。
- `.wild` 编辑、校验、保存、导出工作流。
- AI 对话生成和修改 `.wild` 的工具链。

## 1. 产品定义

WILD 建筑编辑器是一个 AI 辅助的参数化 3D 建筑编辑器。

它不是“一句话生成死模型”的工具，而是一个可以持续编辑的建筑设计工作台：

```text
用户输入自然语言
  -> AI 生成 .wild 初稿或修改建议
  -> 编辑器展示 3D 结果
  -> 用户选择构件、拖拽、调参、确认或撤销
  -> .wild 文件持续作为唯一源文件
  -> wild-core 根据 .wild 确定性重建几何
```

核心原则：

| 原则                     | 说明                                            |
| ------------------------ | ----------------------------------------------- |
| `.wild` 是唯一场景源文件 | 所有建筑结构、材质、行为最终都落到 `.wild` JSON |
| 渲染引擎只复用不重做     | `wild-core` 是现有核心引擎，新系统只接入它      |
| AI 不直接生成三角形      | AI 只生成参数、构件、材质、patch                |
| 用户保留最终决策权       | 批量修改、删除、覆盖场景必须用户确认            |
| 手动编辑不走 Agent       | 拖拽、属性面板、场景树操作直接修改本地状态      |

## 2. 已有资产与边界

当前项目中已有：

```text
wild-lang/              WILD 语言规范、schema、types
wild-core/              现有渲染/重建引擎
wild-viewer/            现有查看器，可作为参考，不作为最终编辑器
lantu/                  示例 .wild 文件
```

### 2.1 必须保留

`wild-core` 的职责保持不变：

```text
输入: Blueprint(.wild JSON)
输出: ReconstructedEntity
      - meshes
      - materialParams
      - boundingBox
      - physics
      - scripts
      - animation
```

现有核心入口：

```ts
parseBlueprint(json: string): Blueprint
reconstructEntity(bp: Blueprint): Promise<ReconstructedEntity>
```

新前端只需要调用这两个入口，然后把 `MeshData` 转成 Three.js 对象。

### 2.2 可以参考但不直接作为最终产品

`wild-viewer` 当前是查看器，不是完整编辑器。它可以作为以下内容的参考：

- 如何加载 `.wild` 文件。
- 如何调用 `parseBlueprint()` 和 `reconstructEntity()`。
- 如何把 `MeshData` 转换成 Three.js `BufferGeometry`。
- 如何处理基础点击交互。

但最终编辑器需要从 0 设计状态管理、组件、AgentBridge、属性面板、构件库、撤销重做等能力。

### 2.3 不允许重做

新项目不做这些事：

- 不重写 `wild-core` 的几何生成算法。
- 不让后端渲染 `.wild`。
- 不让 Agent 输出 Three.js 代码。
- 不把 GLB、OBJ、ThreeJSON 作为主资产格式。
- 不把 UI 状态混入几何语义。

## 5. 核心数据模型

### 5.1 Blueprint

`.wild` 文件本质是一个 JSON Blueprint。

最小结构：

```json
{
  "meta": {
    "version": "1.0",
    "type": "building",
    "name": "未命名建筑"
  },
  "geometry": {
    "elements": []
  },
  "materials": {},
  "behaviors": {}
}
```

标准顶层字段：

| 字段        | 是否必需 | 说明                         |
| ----------- | -------- | ---------------------------- |
| `meta`      | 是       | 版本、类型、名称             |
| `geometry`  | 是       | 构件、模板、实例、批量放置   |
| `materials` | 否       | 材质库                       |
| `behaviors` | 否       | 物理、脚本、动画             |
| `editor`    | 否       | 编辑器私有元数据，不影响渲染 |

### 5.2 Geometry

```json
{
  "geometry": {
    "elements": [],
    "templates": {},
    "instances": [],
    "placements": []
  }
}
```

标准构件类型：

| type          | 用途             |
| ------------- | ---------------- |
| `wall`        | 墙体             |
| `floor`       | 地板/平台        |
| `column`      | 柱子             |
| `beam`        | 梁               |
| `roof`        | 屋顶             |
| `opening`     | 门窗洞口         |
| `stair`       | 楼梯             |
| `furniture`   | 参数化家具和瓦片 |
| `dense_brick` | 高分辨率体素细节 |
| `body`        | 化身身体         |

### 5.3 SceneDocument

前端编辑器内部不只保存 Blueprint，还要保存编辑会话状态。

```ts
interface SceneDocument {
  id: string
  name: string
  revision: number
  blueprint: Blueprint
  dirty: boolean
  savedAt?: number
}
```

`revision` 是前后端协作的关键。每次成功应用 patch 后递增。

### 5.4 Editor Metadata

不影响引擎输出的编辑器信息放在 `blueprint.editor`：

```json
{
  "editor": {
    "version": "1.0",
    "groups": {
      "main_house": {
        "name": "主体建筑",
        "elements": ["wall_front", "wall_back", "main_roof"]
      }
    },
    "view": {
      "camera": {
        "position": [12, 10, 12],
        "target": [0, 1.5, 0]
      }
    },
    "agent": {
      "generatedBy": "wild-server",
      "lastPrompt": "生成一座中式凉亭"
    }
  }
}
```

硬规则：

- `editor` 不能影响 `wild-core` 的重建结果。
- `wild-core` 不能读取 `editor` 字段。
- 导出纯资产时可以剥离 `editor`。

## 6. ScenePatch 协议

所有编辑操作，无论来自用户还是 Agent，都转成 `ScenePatch`。

### 6.1 Patch 基本结构

```ts
interface ScenePatch {
  type: 'scene_patch'
  patch_id: string
  base_revision: number
  source: 'user' | 'agent' | 'system'
  mode: 'apply' | 'proposal'
  requires_confirmation: boolean
  operations: SceneOperation[]
  summary?: string
}
```

### 6.2 操作类型

```ts
type SceneOperation =
  | { op: 'add_element'; element: GeometryElement }
  | { op: 'update_element'; id: string; changes: Record<string, unknown> }
  | { op: 'remove_element'; id: string }
  | { op: 'upsert_material'; name: string; material: MaterialDef }
  | { op: 'add_template'; name: string; template: GeometryElement }
  | { op: 'update_template'; name: string; changes: Record<string, unknown> }
  | { op: 'remove_template'; name: string }
  | { op: 'add_instance'; instance: InstanceRef }
  | { op: 'add_placement'; placement: Placement }
  | { op: 'set_behaviors'; changes: Record<string, unknown> }
  | { op: 'set_editor_meta'; changes: Record<string, unknown> }
```

### 6.3 应用流程

```text
收到 patch
  -> 检查 base_revision 是否等于当前 revision
  -> 校验 operations 格式
  -> 应用到 Blueprint 副本
  -> 运行 Blueprint 校验
  -> 调用 wild-core reconstructEntity 做 smoke test
  -> 成功后替换 SceneStore.blueprint
  -> revision + 1
  -> 渲染刷新
  -> 记录 undo 快照
```

### 6.4 确认规则

| 操作               | 是否需要确认   |
| ------------------ | -------------- |
| 属性面板单字段修改 | 否             |
| 拖拽添加构件       | 否             |
| Agent 添加普通构件 | 可配置，默认否 |
| Agent 批量添加     | 是             |
| 删除构件           | 是             |
| 覆盖整个场景       | 是             |
| 一键整理           | 是             |
| 风格迁移           | 是             |

### 7.2 前端 Store

#### sceneStore

职责：

- 保存当前 `SceneDocument`。
- 应用 `ScenePatch`。
- 调用重建。
- 管理 dirty 状态。

核心状态：

```ts
interface SceneStoreState {
  document: SceneDocument | null
  reconstructed: ReconstructedEntity | null
  validationIssues: ValidationIssue[]
  isReconstructing: boolean
}
```

核心方法：

```ts
loadBlueprint(bp: Blueprint): void
applyPatch(patch: ScenePatch): Promise<void>
reconstruct(): Promise<void>
validate(): ValidationIssue[]
exportWild(): string
```

#### selectionStore

职责：

- 当前选中构件。
- hover 构件。
- 多选。

```ts
interface SelectionStoreState {
  selectedIds: string[]
  hoveredId: string | null
}
```

#### historyStore

职责：

- 撤销。
- 重做。
- patch 历史。

```ts
interface HistoryEntry {
  label: string
  before: Blueprint
  after: Blueprint
  patch: ScenePatch
}
```

#### agentStore

职责：

- 聊天消息。
- WebSocket 状态。
- Agent 正在执行的步骤。
- 等待用户确认的 patch。

### 7.3 渲染适配层

前端不能把渲染转换逻辑散落在 Vue 组件里，必须集中在 `renderer/`。

```text
renderer/wildCoreAdapter.ts
  - import wild-core
  - parseBlueprintText()
  - reconstructBlueprint()

renderer/meshDataToGeometry.ts
  - MeshData -> THREE.BufferGeometry

renderer/materialAdapter.ts
  - MaterialParams -> THREE.Material

renderer/renderEntity.ts
  - ReconstructedEntity -> THREE.Group

renderer/picking.ts
  - raycast result -> elementId
```

重建流程：

```text
SceneStore.blueprint
  -> wildCoreAdapter.reconstructBlueprint()
  -> ReconstructedEntity
  -> renderEntity()
  -> CanvasViewport 替换场景 group
```

MVP 可以整场重建。后期再做局部重建和缓存。

### 7.4 属性面板

属性面板根据选中构件的 `type` 动态渲染。

示例：

| 构件        | 控件                                                  |
| ----------- | ----------------------------------------------------- |
| `wall`      | from/to 坐标、厚度、材质、曲线                        |
| `column`    | base、height、bottomRadius、topRadius、style、材质    |
| `roof`      | roofType、span、depth、height、thickness、tiers、材质 |
| `opening`   | parentWall、from、width、height、style、材质          |
| `furniture` | subtype、position、dimensions、style、材质            |

属性变更时生成 patch：

```ts
{
  op: 'update_element',
  id: 'col_03',
  changes: { height: 3.8 }
}
```

### 7.5 构件库

构件库分两类：

```text
预设构件
  - wall
  - floor
  - column
  - beam
  - roof
  - opening
  - stair
  - furniture

AI 生成模块
  - 凉亭柱网
  - 门楣模板
  - 瓦片 placement
  - 中式屋顶组合
```

拖拽到视口时：

```text
drag start
  -> 创建构件草稿
drop on viewport
  -> 根据拾取点计算位置
  -> 生成 add_element 或 add_instance patch
  -> applyPatch
```

## 8. 后端架构

后端只负责 Agent、会话、保存、校验，不负责渲染。

### 8.1 服务模块

```text
FastAPI app
├── WebSocket Agent API       ← 传输层：收发消息，不碰 AI 逻辑
│   └── ws_heartbeat.py       ← 心跳监控工具类（超时检测 + 回调通知，与具体端点解耦）
├── Agent Service             ← 服务层：管理 Agent 生命周期，对外暴露 query()
├── LangGraph Agent           ← 编排层：定义"思考步骤"（graph + state + prompts）
├── Model Client              ← 工厂层：纯 LLM 初始化，无副作用
├── Tool Layer                ← 工具层：读写场景、生成 patch
├── Scene REST API
├── Validate REST API
├── Session Service
├── Scene Storage
└── Validation Service
```

**Agent 相关模块的分层关系**（从外到内，层层升级）：

```text
api/ws_agent.py          ← 传输：收 WebSocket 消息 → 调 AgentService → 发回前端
    ↓
services/agent_service.py ← 服务：组装 agent/ 各模块，对外暴露 query()，对内决定调用方式
    ↓
agent/graph.py           ← 编排：LangGraph 状态图，load_context → classify → inspect → plan → validate → respond
    ↓
agent/model_client.py    ← 工厂：create_llm()，纯函数，无全局变量
```

**MVP 阶段 `agent_service.py` 内部实现**：

```text
┌─ AgentService.__init__() ──────────────────────────────────────────┐
│                                                                     │
│  create_llm()          ← 调 model_client，拿到裸 LLM               │
│       ↓                                                            │
│  create_agent(          ← LangChain 包装：llm + tools + system_prompt │
│      model=llm,                          → 变成能对话的 agent       │
│      tools=[],                                                      │
│      system_prompt=...                                              │
│  )                                                                  │
│       ↓                                                            │
│  self.agent            ← 复用此实例，query() 每次调 agent.ainvoke() │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**升级路径**：MVP 用 `create_agent` 包一层就够；Phase 4 把 `create_agent` 替换为 LangGraph `graph`，`query()` 内部从 `agent.ainvoke()` 改为 `graph.ainvoke()`，`ws_agent.py` 一行不动。

#### 心跳协议

前端每 15s 发送 ping，后端立即回复 pong。后端通过 `WebSocketHeartbeat` 工具类监控：
空闲超过 90s 未收到任何消息 → 发送 `network_error` → 关闭连接。AI 处理期间
（`is_processing=True`）豁免超时检查。

```json

// 后端 → 前端
{ "type": "pong", "timestamp": 1711234567890 }

// 后端 → 前端（超时通知）
{ "type": "network_error", "error": "心跳超时，连接已断开", "reason": "heartbeat_timeout" }
```

前端连接管理：
- 指数退避自动重连（1s → 2s → 4s → ... → 30s 上限，±25% 抖动）
- 最大重试 10 次，超出后提示用户刷新
- 监听页面可见性变化（`visibilitychange`）：页面恢复可见时立即检测连接并补发心跳，
  解决浏览器后台标签页对 `setInterval` 的节流问题

## 9. Agent 架构

当前阶段使用单 Agent，不使用多 Agent。

### 9.1 原因

当前任务域集中在一份 `.wild` 文件上：

- 生成建筑。
- 添加构件。
- 修改参数。
- 生成模板。
- 给设计建议。
- 校验合理性。

这些不需要多个 Agent 协作。多 Agent 会增加通信、冲突合并、调试和延迟成本。

### 9.2 LangGraph 状态图

```text
用户消息
  -> load_context
  -> classify_intent
  -> inspect_scene
  -> plan_patch
  -> validate_patch
  -> respond
```

节点说明：

| 节点              | 职责                                     |
| ----------------- | ---------------------------------------- |
| `load_context`    | 读取用户消息、场景摘要、当前选择         |
| `classify_intent` | 判断是新建、添加、修改、删除、建议、解释 |
| `inspect_scene`   | 需要时调用读取工具获取元素详情           |
| `plan_patch`      | 生成完整场景或 ScenePatch                |
| `validate_patch`  | 校验 patch 是否符合规则                  |
| `respond`         | 返回自然语言说明和 patch                 |

### 9.3 Agent State

```py
class AgentState(TypedDict):
    session_id: str
    request_id: str
    scene_id: str | None
    scene_revision: int
    user_message: str
    scene_summary: dict
    selection: list[str]
    messages: list[dict]

```

读工具：

| 工具               | 输入          | 输出         |
| ------------------ | ------------- | ------------ |
| `inspect_scene`    | scene_summary | 场景概览     |
| `list_elements`    | type?         | 元素摘要列表 |
| `get_element_info` | element_id    | 元素完整参数 |
| `list_materials`   | 无            | 材质列表     |
| `get_selection`    | 无            | 当前选中元素 |

写工具：

| 工具                     | 输入         | 输出                              |
| ------------------------ | ------------ | --------------------------------- |
| `generate_scene`         | description  | 完整 Blueprint 草稿               |
| `propose_add_element`    | type, params | add_element patch                 |

### 9.5 Agent 行为规则

Agent 必须遵守：

- 所有场景修改都输出 `ScenePatch`。
- 不直接输出 Three.js、HTML、CSS、JS。
- 不生成未知构件类型。
- 不修改 `wild-core`。
- 不生成大段不可校验 JSON。
- 不清楚用户指代时，如果有 selection 就使用 selection，没有 selection 就提问。
- 删除、覆盖、批量整理必须设置 `requires_confirmation: true`。

#### 9.6.3 System Prompt 组装

**位置**：`app/agent/prompts.py`

`build_system_prompt(spec_text)` 将规范文档 + 空间规则 + 工作流程 + 输出格式组装成完整的 System Prompt。

核心规则（铁律）：
1. opening 的 from = [沿墙距离, 距墙底高度, 法向偏移]，禁止使用世界坐标
2. 墙体转角必须共享端点坐标，不得有缝隙
3. 屋顶 span/depth 必须覆盖被支撑墙体范围
4. 楼梯端点高度必须对齐地板或墙体

### 9.7 服务器端错误检测与自动修正流水线（✨ 已实现）

**背景**：LLM 自由调用工具不可控，需要提供确定性保障。为提高 AI 建模结果的稳定性，所有 Validation Tool 按固定流水线顺序执行，不依赖 LLM 自由决定调用顺序。

**设计原则**：
- **确定性流水线**：固定步骤顺序，确保每次校验流程一致
- **前置检测**：Structure → Schema → Reference → Geometry 分层校验
- **自动修正**：检测到可修复问题时自动应用修正算法
- **重检机制**：修正后重新运行相关检测，确保问题真正解决
- **前端可视化**：流水线步骤实时展示在前端页面
- **LangGraph 兼容**：工具和流程设计与 LangGraph 兼容，后续可直接迁移

#### 9.7.1 流水线执行顺序

完整 15 步校验流水线（9 检测 + 6 修正）：

```text
1  validate_blueprint_structure      ← 顶层结构 + ID 唯一性
2  validate_element_required_fields  ← 必填字段 + 枚举合法性
3  validate_reference_integrity      ← 跨构件引用合法性
4  validate_opening_coords           ← 门窗沿墙距离格式
4b validate_opening_fit              ← 开口越界检查（顶点爆炸根源）
5  validate_wall_junctions           ← 墙体转角端点对齐
6  validate_stair_alignment          ← 楼梯端点高度对齐
7  validate_roof_coverage            ← 屋顶覆盖范围
7b validate_element_dimensions       ← 各构件尺寸合理性
8  fix_opening_coords                ← 自动修正门窗 from[0]（如有问题）
8b fix_opening_fit                   ← 自动修正开口越界（如 Step4b 有严重越界）
8c fix_stair_alignment               ← 自动修正楼梯对齐（如 Step6 有问题）
8d fix_element_dimensions            ← 自动修正构件尺寸（如 Step7b 有严重异常）
8e fix_roof_coverage                 ← 自动修正屋顶覆盖（如 Step7 有问题）
8f fix_wall_junctions                ← 自动对齐孤立墙体端点（如 Step5 有问题）
9  validate_collision                ← 碰撞/穿插/悬空/重叠检测
```

**流水线逻辑**：
- 前序步骤失败会跳过后续步骤（如 Step 1 结构校验失败则跳过所有几何校验）
- 修正步骤只在检测到对应问题时才触发
- 每次修正后都会重新运行相关检测步骤验证修正效果

#### 9.7.2 关键错误检测与自动修正

**P0：顶点爆炸防护（最高优先级）**

```python
@tool
def validate_opening_fit(blueprint: dict) -> str:
    """检查 opening 是否超出 parentWall 范围（顶点爆炸根源）"""
    # 规则：沿墙方向 from[0]≥0 且 from[0]+width≤墙长
    #       高度方向 from[1]≥墙底Y 且 from[1]+height≤墙顶Y
    # 返回：❌ 级别错误表示严重越界

@tool
def fix_opening_fit(blueprint: dict) -> str:
    """自动修正超出 parentWall 范围的 opening"""
    # 修正策略：沿墙方向钳位到 [0, 墙长-width]，width 超出时等比例缩小
    #           高度方向钳位到 [墙底Y, 墙顶Y-height]，height 超出时等比例缩小
    # 仅对严重越界（>0.5m）进行修正，保留轻微偏差
```

**P1：尺寸合理性检查**

```python
@tool
def validate_element_dimensions(blueprint: dict) -> str:
    """检查所有构件尺寸是否在合理范围内"""
    # wall 厚度 0.01~5m，长度 0.1~500m，高度 0.1~50m
    # column 半径 0.01~5m，高度 0.1~50m
    # roof span/depth 0.5~500m
    # opening width/height 0.1~20m
    # stair 高差 0.1~50m，水平距离 0.1~100m

@tool
def fix_element_dimensions(blueprint: dict) -> str:
    """自动修正严重异常的构件尺寸"""
    # 修正策略：超出上限 2 倍以上的尺寸才修正
    #           按合理范围上限的 0.9 倍收缩
    #           保留组件比例（如 wall 的厚度与高度比例）
```

**P2：几何对齐修正**

```python
@tool
def fix_wall_junctions(blueprint: dict) -> str:
    """自动对齐孤立的墙体端点，消除墙体转角缝隙"""
    # 修正逻辑：对每个孤立端点，找距离最近的其他墙体端点
    #           若距离在 (0.15m, 1.5m] 范围内，则把孤立端点坐标对齐到目标端点

@tool
def fix_stair_alignment(blueprint: dict) -> str:
    """自动修正楼梯端点高度，使其对齐到最近的地板或墙体高度"""
    # 修正逻辑：收集所有可用参考高度（地板顶面Y、墙体顶部Y、地面Y=0）
    #           将 stair.from.y 对齐到最近的参考高度（容差 0.2m 内才修正）
    #           将 stair.to.y 对齐到下一个更高的参考高度
    #           确保 from.y < to.y（楼梯向上）
```

**P3：屋顶覆盖修正**

```python
@tool
def fix_roof_coverage(blueprint: dict) -> str:
    """自动修正屋顶的 span/depth/position，使其合理覆盖墙体包围盒"""
    # 修正逻辑：计算所有 wall 的 XZ 包围盒
    #           span = wall X 跨度 + 出檐（默认 1.0m 每侧）
    #           depth = wall Z 跨度 + 出檐（默认 1.0m 每侧）
    #           position.x = wall X 中心
    #           position.z = wall Z 中心
    #           position.y 保持不变
```

- 🔧 自动修正的步骤

#### 9.7.5 技术架构优势

**1. 确定性与可重复性**
- 固定流水线顺序，不依赖 LLM 随机性
- 每次校验结果一致，便于调试和问题排查

**2. 分层错误处理**
- 结构错误（Step 1-3）优先检查，避免无效的几何校验
- 几何错误（Step 4-7b）分层检查，问题定位准确
- 自动修正（Step 8-8f）针对性修复，最小化干预

**3. 渐进式严格**
- 仅对 ❌ 级别严���错误自动修正
- ⚠️ 级别警告保留给用户判断
- 保留用户设计意图，避免过度修正

**4. 扩展友好**
- 所有工具都是纯函数，无状态依赖
- 与 LangGraph 兼容，可无缝迁移到图编排
- 新增校验工具只需注册到流水线，无需修改核心逻辑

**5. 用户感知优化**

#### 9.7.6 应用效果

**预防问题**：
1. **顶点爆炸**：通过 `validate_opening_fit` 检测 + `fix_opening_fit` 修正 + 前端 50,000 顶点限制

**用户体验**：
- 流水线步骤可视化，用户了解系统工作状态
- 自动修正减少人工干预，提高建模效率
- 严重错误有明确反馈和占位显示，不影响整体体验

## 10. 校验体系

### 10.1 前端校验

```text

  -> revision 校验

语义校验包括：

后端校验用于提前过滤 Agent 错误，但最终是否应用仍由前端决定。

## 15. 最终架构总结

最终系统由三部分组成：

```text
1. wild-core
   已有渲染/重建引擎。
   不重做，不由 Agent 修改。

2. wild-editor
   从 0 开发的前端编辑器。
   负责 UI、状态、patch、调用 wild-core、渲染 Three.js。

3. wild-server
   从 0 开发的后端 Agent 服务。
   负责理解用户意图，生成 .wild Blueprint 或 ScenePatch。
```

最重要的数据流：

```text
用户操作 / Agent 建议
  -> ScenePatch
  -> SceneStore.blueprint
  -> wild-core.reconstructEntity()
  -> ReconstructedEntity
  -> Three.js 渲染
```

最重要的边界：

```text
Agent 只改参数，不生成几何。
Editor 只管状态，不发明语言。
wild-core 只管重建，不管 UI 和 AI。
.wild 永远是唯一源文件。
```

只要遵守这四条，前端和后端可以从 0 开发，现有渲染引擎可以稳定复用，后续再逐步扩展 MCP、RAG、局部重建和多格式导出。

### 17.2 Function Calling（工具调用）现状说明

**说明**：项目中已经在使用 Function Calling（即 LangChain 的 `@tool` 装饰器机制）。`spatial_tools.py` 里 15 个工具函数注册给 LangChain `create_agent` 后，框架会把它们的 schema 作为 tools 列表传给 LLM，LLM 自主决定调用哪些工具、传什么参数。

**Function Calling 的两个作用**：
1. 让 LLM 决定"何时调用工具、传什么参数"（主要用途）
2. 规范工具输入的格式（副作用）

**当前问题**：小模型的 tool use 可靠性差——不一定会主动调用全部 6 个校验工具，或调用顺序错误，导致带有空间错误的蓝图被直接输出。

**待做优化（服务端驱动的二阶段校验）**：

把"LLM 自愿调用工具校验"改成"服务端强制执行校验"，流程变为：

```text
阶段1：纯生成
  prompt 只聚焦生成，不带工具列表
  → LLM 输出 Blueprint JSON 初稿

阶段2：服务端校验（确定性 Python）
  agent_service 直接调用 spatial_tools 的底层函数（不经过 LLM）
  → 收集所有校验问题

阶段3：LLM 修正（如有问题）
  把校验结果拼入 prompt，让 LLM 针对性修正
  → 输出最终 Blueprint

### 17.3 错误检测与自动修正流水线（已完成）

**背景**：为提高 AI 建模结果的稳定性，需要实现一套完整的错误检测与自动修正系统，防止顶点爆炸、几何错位、尺寸异常等问题。

**决策**：实现服务端驱动的确定性校验流水线，结合前端保护机制，形成多层防护体系。

#### 17.3.1 工具扩展

从最初的 6 个工具扩展到 **15 个纯函数工具**：

**检测工具（9 个）**：
- `validate_blueprint_structure` - 顶层结构完整性
- `validate_element_required_fields` - 必填字段检查
- `validate_reference_integrity` - 引用关系检查
- `validate_opening_coords` - 门窗坐标格式检查
- `validate_opening_fit` - 开口越界检查（顶点爆炸根源）
- `validate_wall_junctions` - 墙体端点对齐检查
- `validate_stair_alignment` - 楼梯高度对齐检查
- `validate_roof_coverage` - 屋顶覆盖检查
- `validate_element_dimensions` - 构件尺寸合理性检查
- `validate_collision` - 碰撞检测

**修正工具（6 个）**：
- `fix_element_dimensions` - 自动修正严重尺寸异常
- `fix_opening_coords` - 自动修正门窗坐标格式
- `fix_opening_fit` - 自动修正开口越界
- `fix_stair_alignment` - 自动修正楼梯高度对齐
- `fix_roof_coverage` - 自动修正屋顶覆盖
- `fix_wall_junctions` - 自动对齐孤立墙体端点

#### 17.3.2 流水线设计

实现 **15 步固定顺序校验流水线**：

```text
Structure → Schema → Reference → Geometry → Fix → Collision → Render
```

关键特性：
1. **确定性执行**：固定步骤顺序，不依赖 LLM 随机调用
2. **分层阻断**：前序步骤失败则跳过后续步骤
3. **自动修正**：检测到可修复问题时自动应用修正算法
4. **重检机制**：修正后重新运行相关检测，确保问题真正解决
5. **前端可视化**：流水线步骤实时展示在前端页面

#### 17.3.3 前端保护机制

**顶点数限制**：
```typescript
const MAX_VERTICES = 50000
if (rawPositions && rawPositions.length / 3 > MAX_VERTICES) {
    // 替换为红色线框占位网格，提示用户构件有问题
}
```

**流水线可视化**：前端实时显示每个校验步骤的状态（✅/⚠️/❌/🔧）

#### 17.3.4 LLM 思考动画

**实现**：LLM 生成期间每隔 8 秒推送一条思考进度消息，旋转显示不同思考状态：
- "理解建筑需求，规划构件组合..."
- "计算空间布局与构件尺寸..."
- "校验开口位置与墙体连接..."
- "优化屋顶覆盖与材料匹配..."

**作用**：让用户在等待期间感知系统工作状态，提升用户体验。

#### 17.3.5 技术优势

1. **防止顶点爆炸**：三层防护（服务端检测 + 自动修正 + 前端限制）
2. **保证几何合理性**：自动对齐墙体端点、楼梯高度、屋顶覆盖
3. **尺寸合规**：检测并修正严重异常的构件尺寸

### 17.3 校验流水线（已完成）及 LangGraph 升级路径

**当前实现**：`agent_service.run_validation_pipeline()` — 服务端驱动的固定顺序 Python 函数调用。

完整流水线顺序（Structure → Schema → Reference → Geometry → Fix → Collision）：

| Step | 函数 | 职责 | 失败时 |
|------|------|------|--------|
| 1 | `validate_blueprint_structure` | 顶层结构 + ID 唯一性 | 后续全部跳过 |
| 2 | `validate_element_required_fields` | 必填字段 + 枚举合法性 | 几何步骤跳过 |
| 3 | `validate_reference_integrity` | parentWall / templateId / constraint target 引用合法性 | 继续执行 |
| 4 | `validate_opening_coords` | 门窗沿墙距离格式 | 触发 Step 8 |
| 4b | `validate_opening_fit` | 开口越界检查（顶点爆炸根源） | 触发 Step 8/8b |
| 5 | `validate_wall_junctions` | 墙体转角端点对齐 | 触发 Step 8f |
| 6 | `validate_stair_alignment` | 楼梯端点高度对齐 | 触发 Step 8c |
| 7 | `validate_roof_coverage` | 屋顶 span/depth 覆盖范围 | 触发 Step 8e |
| 7b | `validate_element_dimensions` | 构件尺寸合理性（0.1~500m 等） | 触发 Step 8d |
| 8 | `fix_opening_coords` + recheck | 自动修正门窗坐标，修正后重跑 Step 4/4b | — |
| 8b | `fix_opening_fit` + recheck | 自动修正开口越界，修正后重跑 Step 4b | — |
| 8c | `fix_stair_alignment` + recheck | 自动修正楼梯高度对齐，修正后重跑 Step 6 | — |
| 8d | `fix_element_dimensions` + recheck | 自动修正严重异常尺寸，修正后重跑 Step 7b | — |
| 8e | `fix_roof_coverage` + recheck | 自动修正屋顶 span/depth/position，修正后重跑 Step 7 | — |
| 8f | `fix_wall_junctions` + recheck | 自动对齐孤立端点（1.5m 以内），修正后重跑 Step 5 | — |
| 9 | `validate_collision` | 碰撞 / 穿插 / 悬空 / 重叠检测 | 继续执行 |

每步结果封装为 `PipelineStepResult`，由 `ws_agent.py` 逐步推送给前端展示。

**当前实现的局限**：
- 线性串行，Step 5/6/7 互不依赖但无法并行
- 条件跳转逻辑写死在 Python if/else 里，不直观
- fix 后的"重试循环"是一次性的，无法多轮迭代直到收敛
- 每个步骤没有独立状态快照，出错时回放困难

**LangGraph 升级方案**：

用 LangGraph 状态图替换 `run_validation_pipeline()`，每个校验步骤变成一个节点，流转逻辑变成边的条件：

```text
                    ┌─────────────────────────────────────────────────┐
                    │                LangGraph 校验图                   │
                    └─────────────────────────────────────────────────┘

[generate]          LLM 生成 Blueprint 初稿
    │
    ▼
[validate_structure]  Step 1：顶层结构
    │ ❌              ✅
    ▼                  ▼
[error_exit]    [validate_schema]   Step 2：必填字段
                    │ ❌            ✅
                    ▼               ▼
              [error_exit]    [validate_reference]  Step 3：引用完整性
                                    │
                                    ▼
                             ┌──────────────┐
                             │  并行执行     │   Step 4/5/6/7
                             │ opening_coords│
                             │ wall_junctions│
                             │ stair_align   │
                             │ roof_coverage │
                             └──────┬───────┘
                                    │ 汇总结果
                                    ▼
                             [fix_if_needed]   Step 8：有问题则修正
                                    │ 修正后
                                    ▼
                             [recheck_opening] 重跑 Step 4
                                    │ 仍有问题且未超过重试上限
                                    │──────────────────► [fix_if_needed]（循环，最多3次）
                                    │ 通过 or 超过上限
                                    ▼
                             [validate_collision]  Step 9：碰撞检测
                                    │
                                    ▼
                              [respond]    组装最终结果，推送前端
```

**升级收益**：
- Step 4/5/6/7 并行执行，校验速度提升约 3~4 倍
- fix → recheck 自动循环，最多重试 3 次直到收敛
- 每个节点有独立状态，出错时可精确定位到哪一步
- 条件边声明式描述，流程可视化，易于扩展新节点

**改动范围**：
- 新增 `app/agent/graph.py` — 定义 LangGraph 状态图
- 修改 `agent_service.py` — 把 `run_validation_pipeline()` 替换为 `graph.ainvoke()`
- `ws_agent.py` 零改动（接口不变）
- `spatial_tools.py` 零改动（纯函数直接复用为节点逻辑）

### 17.4 生成前查询工具（2026-07-22 新增）

**问题**：LLM 在生成 roof/floor 等覆盖类构件时，不知道墙体的实际包围盒尺寸，只能拍脑袋猜测 span/depth。导致 `validate_roof_coverage` 频繁报错（如 span=34 但墙体宽度=60）。

**决策**：新增查询类工具 `get_wall_bounding_box`，让 LLM 在生成 roof/floor 之前先查询墙体 XZ 包围盒，获取精确的目标尺寸。

**实现内容**：
- `spatial_tools.py` 新增 `get_wall_bounding_box` 工具（查询类型：不修改 Blueprint，只返回墙体包围盒信息）
- 返回数据包含：X 方向跨度、Z 方向进深、墙高范围、建议的 roof span/depth/position
- System Prompt 规则 3 强化：**生成 roof 前必须先调用 get_wall_bounding_box**
- 工作流程中新增步骤 2：如有墙体，先查询包围盒再规划构件

**设计理由**：
- 查询工具和检测/修正工具分离——查询不修改 Blueprint，LLM 自主决定如何使用结果
- 流水线仍保留 `fix_roof_coverage` 作为安全网——LLM 即使忽略查询结果，流水线也会自动修正
- System Prompt 中的铁律保证 LLM 行为符合预期，不依赖 LLM "自觉"

### 17.5 wild-core 放置位置与抽取计划（2026-07-23 决策）

**现状**：wild-core（22 个 TypeScript 文件：parser / expander / resolver / 10 种几何构建器 / 材质系统 / 行为脚本）当前放置在 `wild-web/src/wild-core/`，即前端项目源码目录内。仅通过一个适配层 `wild-web/src/renderer/wildCoreAdapter.ts` 被前端其他代码引用。

**当前放置的合理性**：

| 理由 | 说明 |
|---|---|
| 纯计算库 | wild-core 不依赖 Vue、Three.js 或任何前端框架，天然适合在浏览器端跑 |
| 渲染链路全在前端 | 引擎放前端避免了网络往返——不需要把 blueprint 发给服务端算几何再传回来 |
| 隔离良好 | 只有一个入口适配层引用它，依赖链干净：`wild-core → wildCoreAdapter → 前端其他代码` |
| 频繁改动期 | 引擎还在随需求快速迭代，放前端项目内修改和调试更方便 |

**不够理想的地方**：

| 问题 | 说明 |
|---|---|
| 定位不清 | wild-core 是独立核心资产，有自己的类型系统、几何算法、材质系统，和 Vue/Three.js 无关。放在 `src/` 里暗示它是"前端应用代码"，但它实际定位是**独立库** |
| 复用受限 | 目前只能被 wild-web 引用。如果将来有 CLI 工具、自动化测试工具、或换了前端框架，都得从这里拷 |
| 缺少独立工程化 | 独立出去后可以有单独的版本号、单独的单元测试、单独的 CI |

**决策：现阶段保留在 `wild-web/src/wild-core/`，Phase 3 以后抽取为独立 package。**

**目标结构**：

```text
WildAgent/
├── wild-core/          # 独立 package（@wild/core）
│   ├── package.json
│   ├── tsconfig.json
│   ├── src/            # 当前 wild-web/src/wild-core 内容
│   └── tests/
├── wild-web/           # 前端，通过 workspace 或 npm link 引用 @wild/core
└── wild-server/        # 后端（将来也可能引用 wild-core 做服务端校验）
```

**为什么不现在抽**：wild-core 还在随需求频繁改动，抽成独立 package 会多一层 npm link / workspace 的依赖管理成本。等到 Phase 3 以后引擎趋于稳定、改动频率降低后，再做抽取更划算。

**触发抽取的条件**：
1. wild-core 的 API 趋于稳定，改动频率明显降低
2. 需要在 wild-web 之外的另一处引用 wild-core（如 CLI 工具、测试套件）
3. Phase 3（完整编辑工作流）基本完成
