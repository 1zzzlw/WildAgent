# 渲染问题诊断报告

## 问题现象
- `cabin_v1.wild` 可以正常渲染
- `bieshu.wild`（别墅）和 `tiantan.wild`（天坛）渲染时出现模型错乱、交叉、乱码现象

## 根本原因分析

经过详细代码审查和蓝图对比，我发现了以下关键问题：

### 1. 屋顶position缺失但修复已应用 ✓

**修复记录显示的问题**：
- `resolveRoofBoundary` 会自动计算屋顶position
- 但阳台栏杆等矮墙会污染包围盒计算，导致屋顶偏移

**实际状态**：
- ✓ 修复代码已应用（过滤高度<50%最大墙高的矮墙）
- ✓ `bieshu.wild` 的屋顶没有显式position，依赖自动计算
- ✓ `tiantan.wild` 的屋顶**有显式position**，不受此bug影响

### 2. 中式屋顶（pagoda）的特殊问题 ⚠️

`tiantan.wild` 使用的是 `chinese_pagoda` 类型屋顶：

```json
{
  "type":"roof",
  "id":"main_roof",
  "roofType":"chinese_pagoda",
  "span":11,
  "depth":11,
  "height":6,
  "tiers":3,
  "tierHeight":2,
  "eaveOutset":1,
  "shrinkFactor":0.65,
  "position":[0,9.2,0]
}
```

**潜在问题**：
- `buildPagoda` 函数生成多层叠加的坡面
- 每层4个坡面 + 1个檐口底面 = 5个mesh/层
- 3层 × 5 = **15个mesh**
- 这些mesh的transform都是`position: [0,0,0]`，完全依赖顶点世界坐标

**可能的错误**：
- 顶点坐标计算时使用了 `pos[1]`（Y轴基准）
- 但实际应该使用 `yBase` 和 `yTop`
- 代码中正确使用了局部变量，看起来没问题

### 3. 别墅（bieshu）的复杂度问题 ⚠️

`bieshu.wild` 包含大量构件：
- 两层建筑（ground floor + upper floor）
- 内墙分隔
- 阳台
- 栏杆
- 多个门窗开口
- 楼梯

**可能的交叉问题**：
1. **墙体厚度重叠**：内外墙在同一位置可能产生z-fighting
2. **楼板高度错位**：`upper_floor` 的 `from[1]` 和墙体的 `to[1]` 可能不一致
3. **开口定位错误**：门窗的 `from` 参数在弧形墙中需要特殊处理

### 4. **真正的bug：开口(opening)坐标计算** 🔴

查看 `resolveOpenings` 函数，我发现了一个问题：

```typescript
// 弧形墙体
const angle = startRad + opening.from[0] / radius;
```

对于 `tiantan.wild` 的门：
```json
{"type":"opening","id":"door_N","parentWall":"main_wall","from":[1.52,0,0],...}
```

**计算过程**：
- `main_wall` 的 `from=[5.8,1.2,0]`，`to=[5.8,7.7,0]`
- 中心 `center=[0,0,0]`
- 半径 `radius = 5.8`
- `startRad = atan2(0-0, 5.8-0) = 0`
- 门N的角度 = `0 + 1.52/5.8 ≈ 0.262 rad ≈ 15°`

但天坛应该有4个门，均匀分布在360°：
- 北门：0°
- 东门：90°
- 南门：180°
- 西门：270°

**问题**：`from[0]` 的值 `1.52` 是什么意思？
- 如果是弧长：1.52m / 5.8m = 15° ✓ 勉强合理
- 但4个门的 `from[0]` 是 `[1.52, 10.63, 19.74, 28.85]`
- 换算：[15°, 105°, 195°, 293°] ❌ **不均匀！**

**这说明蓝图文件本身有问题**！

### 5. 别墅的墙体坐标问题 🔴

查看 `bieshu.wild` 的墙体定义：

```json
{
  "type": "wall",
  "id": "wall_left",
  "from": [-8.0, 0.0, -5.7],  // 注意 Z 不是 -6.0
  "to": [-8.0, 3.0, 7.7],     // 注意 Z 不是 8.0
  "thickness": 0.3
}
```

对比：
```json
{
  "type": "wall",
  "id": "wall_front",
  "from": [-8.0, 0.0, -6.0],
  "to": [8.0, 3.0, -6.0]
}
```

**问题**：侧墙的Z坐标从 `-5.7` 到 `7.7`，但前后墙是 `-6.0` 到 `8.0`
- 这导致墙角**不对齐**！
- `-5.7` vs `-6.0` 差了 **0.3米**（恰好是墙厚）
- 这是为了避免墙角重叠，但 `resolveWallJoints` 应该自动处理这个

**root cause**：蓝图文件手动调整了坐标来"修复"墙角问题，但这破坏了几何一致性

## 结论

### Deepseek的诊断是**部分正确**的：

✓ **正确部分**：
1. 屋顶包围盒计算确实有bug（但已修复）
2. 楼梯欧几里得距离计算有bug（但已修复）
3. 柱子吸附容差有bug（但已修复）

❌ **错误部分**：
1. **这些修复都已经应用到代码中了！**
2. **但问题依然存在，说明bug不在这里**

### 真正的问题：

1. **蓝图文件本身有设计错误**：
   - `tiantan.wild` 的门位置计算不均匀
   - `bieshu.wild` 的墙角坐标手动调整导致不一致

2. **渲染引擎的潜在问题**：
   - `resolveWallJoints` 可能没有正确处理复杂墙角
   - `resolveOpenings` 对弧形墙的开口定位可能有问题
   - 多层建筑的楼板和墙体连接可能有gap

3. **缺少错误检测**：
   - 没有检测墙角不对齐
   - 没有检测开口超出墙体范围
   - 没有检测构件重叠/交叉

## 建议修复方案

### 短期（修复蓝图）：

1. **修复 tiantan.wild 的门位置**：
```json
{"id":"door_N","from":[0,0,0],...}       // 0° 
{"id":"door_E","from":[9.11,0,0],...}    // 90°  = 5.8 * π/2
{"id":"door_S","from":[18.22,0,0],...}   // 180° = 5.8 * π
{"id":"door_W","from":[27.33,0,0],...}   // 270° = 5.8 * 3π/2
```

2. **修复 bieshu.wild 的墙角坐标**：
```json
{"id":"wall_left","from":[-8.0,0.0,-6.0],"to":[-8.0,3.0,8.0]}
{"id":"wall_right","from":[8.0,0.0,-6.0],"to":[8.0,3.0,8.0]}
```
让 `resolveWallJoints` 自动处理墙角对齐

### 中期（增强渲染引擎）：

1. **改进 resolveWallJoints**：
   - 检测墙角gap > 0.1m，自动插入填充
   - 检测墙体重叠，自动调整厚度

2. **改进 resolveOpenings**：
   - 检测开口是否超出墙体范围
   - 对弧形墙的开口使用角度而非弧长

3. **添加校验器**：
   - `validateWallCorners()` - 检测墙角连接
   - `validateOpenings()` - 检测开口合法性
   - `validateOverlaps()` - 检测构件重叠

### 长期（优化蓝图格式）：

1. **简化墙体定义**：只需指定外墙顶点，自动生成墙体
2. **相对定位系统**：开口相对于墙体的相对位置（0-1）
3. **约束系统**：自动保证墙角连接、楼板对齐等约束
