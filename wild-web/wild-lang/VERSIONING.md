# 原语语言版本管理 v1.0

本文档定义原语语言的版本标识、兼容性规则、引擎版本验证逻辑和语言规范修订流程。原语语言遵循严格的语义化版本控制和“永不删除”的向后兼容承诺。

---

## 一、版本号规范

原语语言的版本号遵循语义化版本规范 (Semantic Versioning)，格式为 `MAJOR.MINOR.PATCH`。

### 1.1 版本号字段

| 版本段 | 含义 | 示例 |
|--------|------|------|
| **MAJOR** | 主版本号。增加不兼容的语法变更时递增 | `1.0.0` → `2.0.0` |
| **MINOR** | 次版本号。增加向后兼容的新构件、新字段、新效果层时递增 | `1.0.0` → `1.1.0` |
| **PATCH** | 修订号。对语言规范文本进行澄清、修正错误描述时递增 | `1.0.0` → `1.0.1` |

### 1.2 蓝图中如何声明版本

每份 `.wild` 蓝图在 `meta.version` 字段中声明其使用的语言版本：

```json
{
  "meta": {
    "version": "1.0",
    ...
  }
}
蓝图只声明 MAJOR.MINOR，不包含 PATCH。修订号级别的变更不影响蓝图语法，因此蓝图无需关心 PATCH 版本。

二、向后兼容承诺：“永不删除”原则
原语语言规范遵循严格的向后兼容规则。这是原荒协议对数字资产“永远存在”承诺的技术基石。

2.1 始终向后兼容的变更
以下变更可在任何时候在新版本中引入，无需升级 MAJOR 版本号：

增加新的构件类型（如 v1.1 新增 furniture）

为已有构件增加新的可选字段

增加新的效果层类型

扩展已有枚举值（如为 style 增加新选项）

增加新的交互指令

增加新的事件类型

2.2 绝对禁止的破坏性变更
以下变更在原语语言规范中永远不会发生。这不是建议，而是原语语言对全世界的不可撤销承诺：

删除或重命名任何已有的构件类型、字段、效果层或指令

修改已有字段的类型定义（如将 roughness 从 number 改为 string）

修改已有枚举值的语义（如将 "doric" 的含义从多立克柱式改为别的）

修改 baseColor 的标准光照条件

2.3 为什么“永不删除”至关重要
如果一份 v1.0 蓝图在 v2.0 引擎中无法重建，那么所有基于 v1.0 创建的数字资产都会在未来失效。这意味着创作者在发布作品时，必须信任“未来的语言版本不会背叛我”。

原语语言的“永不删除”原则将这份信任变为了一个可验证的密码学承诺。任何人都可以在任何时候拿出任何一份 .wild 文件，用当时的最新引擎版本重建，并验证输出是否与发布时一致。

三、引擎版本验证
原语引擎在解析蓝图时，必须执行版本验证。

3.1 验证逻辑
text
1. 读取蓝图的 meta.version 字段
2. 将该 MAJOR.MINOR 版本号与引擎支持的最高语言版本进行比较
3. 如果蓝图版本 ≤ 引擎版本 → 解析继续
4. 如果蓝图版本 > 引擎版本 → 拒绝解析，返回错误
3.2 引擎版本与语言规范版本的关系
引擎版本号独立于语言规范版本号。引擎可以有自己的发布节奏。但是，引擎必须在内部声明它支持的语言规范最高版本。

示例：

引擎版本	支持的语言规范最高版本
wild-core v1.0.0	原语语言 1.0
wild-core v1.1.0	原语语言 1.1
wild-core v2.0.0	原语语言 2.0
任何引擎实现都必须保留所有历史语言版本的解析能力。引擎内部为每个 MAJOR.MINOR 语言版本维护一个解析器分支。

3.3 不同引擎实现之间的互操作
两个客户端使用不同的引擎实现（如 wild-core 和社区开发的 wild-engine-rs），只要它们都通过了相同语言版本的合规性测试，就可以互操作。它们重建同一份蓝图所产生的网格完全一致。

四、语言规范修订流程
原语语言规范不是由任何单一个体或组织控制的。修订需经过原荒协议的社区治理流程。

4.1 提案与投票
提案：任何社区成员可提交语言规范修订提案，包含变更的完整描述和动机。

讨论期：提案在社区论坛公开讨论。

投票：土地持有者（1 地 1 票）对提案进行投票。

通过阈值：提案需达到法定参与率和多数同意。

4.2 发布
通过后的提案由原荒核心维护者多签小组发布新的语言规范版本。

新规范发布在 wild-lang 仓库中，附带变更说明。

合规性测试套件同步更新，加入新构件或新字段的测试蓝图。

4.3 引擎实现的跟进
社区引擎开发者（包括 wild-core 维护者）在语言规范更新后自主跟进，实现新语法。引擎的合规性由测试套件自动验证，无需任何中心机构“认证”。

五、版本兼容性示例
场景	引擎行为
蓝图 v1.0 + 引擎支持 v1.0	✅ 解析，正常重建
蓝图 v1.0 + 引擎支持 v1.1	✅ 解析，正常重建（引擎向后兼容）
蓝图 v1.0 + 引擎支持 v2.0	✅ 解析，正常重建（引擎向后兼容）
蓝图 v2.0 + 引擎支持 v1.0	❌ 拒绝解析。提示用户升级引擎
蓝图 v1.0（含 furniture 构件）+ 引擎支持 v1.0	❌ 拒绝解析。furniture 构件在 v1.1 才引入
蓝图 v1.1（含 furniture 构件）+ 引擎支持 v1.1	✅ 解析，正常重建
